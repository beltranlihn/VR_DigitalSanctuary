# Unreal Engine 5.8 via MCP — operational guide

We drive Unreal from the native UE 5.8 **ModelContextProtocol** plugin, registered as the MCP server **`unreal`** (HTTP, localhost:8000/mcp). It uses the Tool-Search pattern: three top-level tools expose everything.

> ## 🔴 Contexto del proyecto — lo que cambia todas las respuestas
> **Target: Meta Quest 3 STANDALONE (APK Android). NO es PC VR.** Corre el **renderer MÓVIL**. Obra sentada, 15 min, single-user, estética Turrell (luz de color en el aire, vacíos oscuros). **Todo horneado** — Lumen/Nanite/VSM/DistanceFields no corren. Presupuesto: **72 fps = 13.9 ms**. Fill-rate bound, no geometry bound.
> **Antes de tocar config, materiales o luces → [materials-vr.md](references/materials-vr.md) y [lighting-quest.md](references/lighting-quest.md).** La mitad de lo que se lee por internet asume deferred y PC.

> ## 🔴 REGLA DE ORO DE TOKENS: nunca traigas un output MCP gigante
> Medido acá: `describe_toolset` = **72k chars** · `find_nodes` sin filtro = **146k** · `get_connected_subgraph` = **1.7M**.
> **Filtro específico siempre. `node_class` siempre. Si ya se volcó a archivo → procesarlo con PowerShell/Grep, no leerlo. Si hay que leerlo entero → subagente.** Ver [workflow.md](references/workflow.md).

## Session startup (check this first)
MCP links are established when Claude starts, so **Unreal must already be running before the Claude session begins** (opening the project auto-starts the server — Auto Start Server is enabled on port 8000). If Unreal wasn't up at launch, the `unreal` tools won't exist and no amount of opening it now will attach them — the user must restart Claude.

At the start of any Unreal session, verify the link cheaply before promising anything: call `SceneTools.get_current_level`. If the tools are missing or the link dropped mid-session (observed: it does NOT auto-recover even though the port still answers), the fix is **restart Claude, leave Unreal open**. Confirm the editor is alive with `Get-Process UnrealEditor*` / port 8000 before blaming the plugin.

## How to call anything
- `mcp__unreal__list_toolsets` — list the ~19 toolsets.
- `mcp__unreal__describe_toolset` `{toolset_name}` — ⛔ **NO USAR.** BlueprintTools son ~72k chars. Está todo destilado en [references/toolsets.md](references/toolsets.md). Solo describir en vivo si una tool NO está ahí.
- `mcp__unreal__call_tool` `{toolset_name, tool_name, arguments}`:
  - ⚠ **`toolset_name` exige el PATH COMPLETO**: `editor_toolset.toolsets.blueprint.BlueprintTools`, `...scene.SceneTools`, `...object.ObjectTools`, `...asset.AssetTools`, `...primitive.PrimitiveTools`. Excepciones de otro namespace: **`EditorToolset.LogsToolset`**, `EditorToolset.EditorAppToolset`. El nombre corto **falla** ("Toolset not found").
  - **`tool_name` sí va corto** (después del último punto).
  - Refs: `{"refPath": "/Game/..."}` o `/Script/Module.Class`.

## The core workflow (Blueprints)
1. **Create / locate**: `BlueprintTools.create` (asset_type ref `/Script/Engine.Actor`) → `list_graphs` gives `:EventGraph`, `:UserConstructionScript`, and any function graphs.
2. **BEFORE editing an existing BP: READ IT.** `read_graph_dsl` + `list_variables`. Never edit blind.
3. **Discover nodes**: `find_node_types` (graph, **specific** `type_id_filter`, `context_pins:[]`) → then `get_node_type_pins` for exact pin names/indices. See [references/nodes.md](references/nodes.md) for verified ones.
4. **Build**:
   - **NEW / empty graph** → `write_graph_dsl` (S-expr). See [references/dsl.md](references/dsl.md).
   - **EXISTING graph the user or a tool already built** → **surgical node ops** (`create_node`, `connect_pins`, `set_pin_value`, `delete_node`), guided by `get_node_infos`. Do NOT re-`write_graph_dsl` an existing event — it DUPLICATES it. See [references/gotchas.md](references/gotchas.md).
5. **Compile OFTEN — no solo al final.** `compile_blueprint` es un **detector de errores**, no un trámite de cierre: compilá después de cada cambio significativo (nodos, variables, y **siempre** después de tocar un CDO por propiedades). Un cambio de CDO sin compilar se lee de vuelta bien y falla en runtime — ver gotchas. Después: **Save** `AssetTools.save_assets`.
6. **Place in level**: `SceneTools.add_to_scene_from_asset`.

## Golden rules (full list in references/gotchas.md)
- Read an existing graph before touching it; edit surgically, verify with `read_graph_dsl`/`get_node_infos` after.
- `read_graph_dsl` inlines pure nodes and can mislabel by name-collision — confirm suspicious nodes with `get_node_infos` before assuming a bug.
- Components go on the Blueprint's **CDO** (`get_default_object`), not the asset path.
- Bind a runtime delegate with its **`Assign<Delegate>`** node — it auto-generates a correctly-typed event.
- Don't trust web docs for exact node ids/pins — the live server is ground truth.

## Per-Blueprint progress tracking — 🔴 REGLA (pedido explícito del usuario 2026-07-20)
🗺️ **Índice maestro de TODOS los Blueprints del proyecto: [`blueprints/_INDEX.md`](blueprints/_INDEX.md)** — el mapa de un vistazo (qué es cada BP, dónde está, para qué sirve, estado, y link a su tracker). **Empezá por ahí** para ubicarte. Al crear/modificar un BP, actualizá **su fila en el índice** Y su tracker.

For each BP we work on, keep a tracker at `blueprints/<BP>.md`. **Es OBLIGATORIO, no opcional:**
- **ANTES de ajustar un BP → leer su tracker primero.** Si la info que necesitás (rol de una variable, estructura de un grafo) ya está ahí, NO re-leas el grafo. Solo leé el grafo si el tracker no lo cubre — y en ese caso, documentalo al terminar.
- **DESPUÉS de ajustar un BP → actualizar el tracker** con un **registro de qué hace CADA variable** (su objetivo, no solo el nombre), la estructura del/los grafo(s) (orden del pipeline), y qué se ajustó y por qué. El objetivo es que la próxima vez sepas **dónde buscar sin releer el grafo entero**.
- Modelo de referencia bien documentado: `blueprints/BP_BreathSensor_V2.md` (registro de variables por categoría + estructura de `Step` paso a paso + qué palanca ajusta qué comportamiento). Copiar ese nivel de detalle.
- Secciones sugeridas: Purpose / Status / **Registro de variables (por categoría, con rol)** / **Estructura de grafos (orden del pipeline)** / Session log / TODO / Open questions.
